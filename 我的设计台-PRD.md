# 我的设计台 PRD

## AI 速读卡
- 产品一句话：我的设计台是一款 AI 生图模板管理与可编辑复用工具，让用户把 GPT Image 2 生成的常用图片保存为可直接改字导出的模板。
- 核心循环：导入常用图 -> 标记可编辑区域 -> 保存模板 -> 修改日期/地点/文字 -> 预览 -> 导出。
- 目标平台：桌面优先 Web 工具，移动端只做查看和轻量改字。
- 硬约束：P0 不重新生图；可编辑内容必须以文字层、遮盖层、变量字段保存。
- 推荐默认：本地优先、单用户、无需账号；Prompt 作为模板溯源信息保存。
- 发挥空间：批量替换日期、自动文件命名、历史版本、智能识别文字区域。
- P0 验收：导入一张海报，遮住旧日期，放入新日期文字层，保存模板，重开后改日期并导出 PNG。
- 最容易翻车：把产品做成完整 Photoshop 或生图平台，偏离“高频模板快速复用”。
- 超预期机会：一次填 12 个月日期，自动导出 12 张同风格海报。

## 第一章：产品概述
我的设计台是一款 AI 生图模板复用工具，让经常使用 GPT Image 2 生成宣传图、海报、封面的人，能够把高频图片保存成可编辑模板，而无需每次重新生图或打开复杂设计软件。

### 1.1 差异化对比表
| 功能 | 竞品或替代方式 | 我的设计台 | 实现方式 |
|---|---|---|---|
| 保存常用 AI 生图 | 文件夹只保存最终图片 | 保存图片、Prompt、模型备注、可编辑层 | 模板数据 + 图片资产 + 元数据 |
| 高频改字 | Photoshop/Figma 功能强但流程重 | 打开模板直接改日期、地点、标题 | 文字层 + 字段面板 |
| 处理压平图片文字 | 普通图片查看器无法编辑 | 用遮盖层盖住旧文字，再叠加新文字层 | 遮盖层、底色块、可编辑文字层 |
| Prompt 复用 | 笔记软件里零散记录 | Prompt 跟图片模板绑定 | 模板详情页保存 Prompt |
| 批量月度海报 | 手动复制、重复修改 | 后续支持变量批量导出 | 字段表 + 批量渲染 |

说明：这里的“竞品”是基于常见工作流的产品推断，未做逐项实时竞品验证。

### 1.2 三类用户画像
| 用户 | 核心目标 | 对现有工具最大不满 | 愿意切换的功能 |
|---|---|---|---|
| 个人运营/自媒体 | 每月快速改宣传图日期和活动信息 | 每次找旧图、找 Prompt、重新调样式 | 打开模板直接改字段 |
| 小团队内容负责人 | 把常用海报交给非设计同事复用 | 设计源文件门槛高，容易误改版式 | 锁定底图，只开放可改字段 |
| AI 生图重度用户 | 沉淀高频 GPT Image 2 图片风格 | 成品图和 Prompt 分散保存 | 图片、Prompt、模板层统一归档 |

### 1.3 可行性边界
| 在范围内，及原因 | 明确排除在外，及原因 |
|---|---|
| 导入 GPT Image 2 生成的 PNG/JPG 作为模板底图 | P0 不调用 GPT Image 2 再生成，避免浪费时间和成本 |
| 保存 Prompt、模型名、风格备注，作为溯源信息 | P0 不承诺自动还原生图模型内部参数 |
| 在图片上添加、编辑、拖动文字层 | 不直接把压平图片里的旧文字识别成真正文字 |
| 用遮盖层覆盖旧日期、地点、价格，再叠加新文字 | P0 不做复杂 AI 修图、智能擦除、背景重绘 |
| 导出 PNG/JPG 和模板备份 JSON | P0 不做多人协作、云素材库、模板市场 |
| 桌面浏览器完整编辑 | 移动端不做精细拖拽编辑，避免触控误操作 |

### 1.4 约束分层
| 硬约束 | 推荐默认 | 发挥空间 |
|---|---|---|
| 核心流程必须是“直接改模板并导出”，不是重新生图 | 单用户本地优先，先用浏览器本地存储 | 后续可加云同步 |
| 可改内容必须结构化保存为文字层、遮盖层、变量字段 | 先支持文字、矩形遮盖、图片底图三类图层 | 可增加二维码、贴纸、图标层 |
| 预览和导出必须一致 | 用同一份画布数据渲染预览和导出 | 可做导出前质量检查 |
| Prompt 只作为模板资产信息保存 | 记录 GPT Image 2、正向 Prompt、备注 | 后续可加入“复制 Prompt”或“重新生成”入口 |
| 不误导用户认为压平文字能天然编辑 | 首次模板化时需要手动遮盖旧文字 | 后续可做 OCR 辅助识别 |

## 第四章：超越竞品的差异化功能

### 第 4.1 节：把 AI 生图结果变成可复用模板
1. 竞品为什么没有这个功能：通用设计工具以自由编辑和素材体系为中心，AI 生图工具以生成新图为中心，两者都不天然围绕“我每月只改几个字”的复用场景组织。
2. 本产品如何实现：导入生图成品作为锁定底图，再叠加可编辑文字层、遮盖层和变量字段；Prompt 只做溯源，不强迫用户重新生成。
3. 交互流程：
```text
导入 GPT Image 2 成品图
    |
    v
遮盖旧日期/地点
    |
    v
添加可编辑文字层
    |
    v
绑定字段：日期、地点、时间
    |
    v
保存为模板，下次直接改字段导出
```
4. 风险与应对：如果原图旧文字区域背景复杂，遮盖层可能突兀；P0 提供颜色吸管、透明度和遮盖层调整，P2 再考虑 AI 擦除。

### 第 4.2 节：字段面板驱动图片改字
1. 竞品为什么没有这个功能：传统设计工具主要按图层编辑，非设计用户需要理解图层、选区、字体；字段化不是默认工作流。
2. 本产品如何实现：把文字层绑定到字段，用户不需要点中画布也能在右侧表单改“活动日期”“地点”“时间”。
3. 交互流程：
```text
打开模板
    |
    v
右侧字段表单显示常改内容
    |
    v
用户改“活动日期”
    |
    v
画布对应文字层实时更新
    |
    v
导出
```
4. 风险与应对：字段和图层绑定可能断开；提供“未绑定字段”提示和一键绑定当前选中文字层。

### 第 4.3 节：月度批量复用
1. 竞品为什么没有这个功能：多数工具把每次设计当作独立文件处理，不把“周期性重复生产”作为核心对象。
2. 本产品如何实现：在 P1 支持表格式变量输入，一行变量生成一张图片。
3. 交互流程：
```text
选择模板
    |
    v
填写 12 行日期和地点
    |
    v
系统逐张套用变量
    |
    v
预览缩略图
    |
    v
批量导出 ZIP
```
4. 风险与应对：批量导出可能内存压力大；采用逐张渲染、分批下载，避免一次性渲染全部大图。

## 第六章：技术架构
```text
+----------------------------------------------------------------+
| UI 层：模板库、编辑台、字段面板、图层面板、导出弹窗              |
| 职责：接收用户操作，展示当前模板状态                            |
+----------------------------------------------------------------+
| 状态层：当前模板、选中图层、字段值、撤销栈、导出状态              |
| 职责：保证字段、图层、画布预览同步                              |
+----------------------------------------------------------------+
| 渲染层：画布预览渲染、导出渲染                                  |
| 职责：用同一份图层数据生成屏幕预览和最终图片                    |
+----------------------------------------------------------------+
| 存储层：模板 JSON、本地图片资源、历史版本                         |
| 职责：保存和恢复完整模板，不丢图、不丢字段、不丢 Prompt           |
+----------------------------------------------------------------+
```

| 库名 | 用途 | 为什么优于替代方案 | 大致包体积 |
|---|---|---|---|
| React 或 Vue | 构建 Web 工作台界面 | 任选现有项目栈即可，重点是状态清晰 | 未知 |
| Konva.js 或 Fabric.js | 画布选中、拖拽、文字层、导出 | 比原生 Canvas 更快完成图层编辑体验 | 未知 |
| IndexedDB 封装库 | 保存图片资源和模板数据 | localStorage 不适合保存大图片和版本数据 | 未知 |
| JSZip | P1 批量导出 ZIP | 比手写打包更稳定 | 未知 |

最大架构风险：画布预览和导出结果不一致。应对方式：预览和导出必须读取同一份 `layers` 数据，不能各写一套渲染逻辑。

可替换技术原则：推荐 Web + Konva.js/Fabric.js + IndexedDB；如果项目已有 Canvas/SVG 编辑框架，可以沿用。不可变的是：模板必须保存底图、遮盖层、文字层、字段绑定、Prompt 信息，并且导出必须与预览一致。

## 第九章：开发优先级
- P0 - 没有这个，产品根本无法使用。交付标准：导入图片创建模板；支持文字层、遮盖层、字段绑定；保存模板；重新打开；修改字段；导出 PNG；导出 JSON 备份。
- P1 - 没有这个，用户第一次体验后不一定持续回来。交付标准：历史版本、模板复制、JPG 导出、批量字段导出、颜色吸管、撤销重做。
- P2 - 有了这个，用户会愿意推荐给别人。交付标准：OCR 辅助识别文字区域、智能安全区、二维码层、字体管理、WebP 导出。
- P3 - 有了这个，一部分用户会付费或强烈倡导。交付标准：云同步、团队共享、模板权限、AI 擦除旧文字、可选 GPT Image 2 再生成入口。

| 指标名称 | 目标值 | 测量方法 | 劣化阈值 |
|---|---:|---|---:|
| 打开包含 50 个模板的模板库 | <= 1000ms | 浏览器 Performance 记录首屏可交互 | > 2000ms |

| 保存模板 | <= 500ms | 点击保存到成功提示 | > 1200ms |
| 导出 1080x1350 PNG | <= 3000ms | 点击导出到文件可下载 | > 6000ms |
| 撤销/重做响应 | <= 100ms | 快捷键触发到画布更新 | > 250ms |
| 模板 JSON 备份大小 | <= 2MB，不含图片 | 读取导出 JSON 文件大小 | > 5MB |
| 编辑时帧率 | >= 45fps | Chrome Performance 测量 | < 30fps |

## 第十一章：开发者交接说明

### a) 实现顺序建议
你先实现模板数据模型和画布渲染，再实现图层编辑，最后实现模板库和导出。原因是这个产品的核心是“同一份模板数据能被保存、恢复、预览、导出”，不是页面数量。

推荐顺序：
1. 画布数据结构：底图层、文字层、遮盖层、字段。
2. 画布渲染：用固定测试数据渲染出海报。
3. 图层编辑：选中、拖动、修改文字、修改遮盖层。
4. 字段面板：字段改动能同步到文字层。
5. 本地保存：刷新后模板可恢复。
6. 导出 PNG 和 JSON。
7. 再做模板库、历史版本、批量能力。

### b) 最可能导致返工的三个决策
| 决策 | 安全默认选择 | 什么时候需要改变 |
|---|---|---|
| 是否重新调用 GPT Image 2 | P0 不调用，只保存 Prompt | 用户明确要求一键再生成 |
| 是否自动识别旧文字 | P0 不自动识别，手动遮盖旧文字 | 用户已有大量旧图且模板化成本太高 |
| 是否做云同步账号 | P0 本地优先，无账号 | 用户要多设备、团队共享或长期备份 |

### c) 哪里要严格，哪里可以灵活
| 章节 | 约束或建议 | 原因 |
|---|---|---|
| 第一章产品边界 | 约束 | 不允许偏成生图平台或复杂修图软件 |
| 第二章布局 | 建议 | 可按现有设计系统调整，但必须保留模板库、画布、字段面板 |
| 第三章核心模块 | 约束 | P0 必须完成模板复用闭环 |
| 第五章数据模型 | 约束 | 数据不稳定会导致模板无法长期复用 |
| 第六章技术架构 | 建议 | 可替换具体库，但预览和导出必须同源 |
| 第七章交互细节 | 建议 | 可优化体验，但不要增加学习成本 |
| 第八章导出 | 约束 | PNG 和 JSON 是 P0 必需输出 |
| 第九章优先级 | 约束 | 不要把 P2/P3 功能塞进 P0 |

### d) 已知的未知项
- 此处未解决：用户已有的历史图片中，旧日期/地点所在区域是否多为纯色背景。如果是复杂背景，P0 的遮盖层效果可能不够自然。
- 此处未解决：用户是否需要在不同电脑之间同步模板。默认先做本地单用户，避免账号系统拖慢第一版。
- 此处未解决：最终是否需要直接接入 GPT Image 2。当前产品判断是不进入 P0，因为用户明确不希望频繁再生成。
- 此处未解决：用户常用输出平台是朋友圈、小红书、公众号、线下打印还是全部都有。默认先支持自定义画布尺寸和 PNG 导出。
