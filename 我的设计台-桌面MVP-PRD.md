# 我的设计台桌面 MVP PRD

## AI 速读卡
- 产品一句话：我的设计台是一款桌面端本地海报模板复用工具，让运营和内容人员把项目根目录里的常用活动海报、课程表、导师介绍图变成可改字段、可导出、可备份的模板。
- 核心循环：导入常用底图 -> 识别图片类型和可变字段 -> 创建模板 -> 添加遮盖和文字字段 -> 保存 -> 改字段 -> 导出图片或备份。
- 目标平台：桌面浏览器 Web 工具；移动端不进入本版本范围。
- 硬约束：不做营销页，不做移动端编辑，不重新生图，不上传用户图片到服务器。
- 推荐默认：三栏工作台，本地 IndexedDB 保存，预览和导出共用同一份图层数据；模板库默认支持横版海报、竖版海报、课程表、导师介绍四类。
- 发挥空间：空态引导、右侧说明、按钮状态、快捷键提示、画布操作细节、默认字段推荐可继续打磨。
- P0 验收：用户在桌面端从常用素材中导入一张图，完成模板创建、字段修改、保存、重新打开和 PNG 导出。
- 最容易翻车：做成复杂设计软件、移动端适配项目或 AI 生图平台。
- 超预期机会：首次空态直接教会用户三步复用流程，并把本地保存和备份风险说清楚。

## 第一章：产品概述
我的设计台是一款桌面端 AI 生图成品复用工具，让经常制作海报、封面、活动图的运营和内容人员能够把常用图片沉淀为可编辑模板，而无需每次重新生图、重新排版或打开复杂设计软件。

### 1.1 差异化对比表
| 功能 | 竞品或替代方式 | 本产品 | 实现方式 |
|---|---|---|---|
| 复用压平成品图 | 文件夹只能保存最终图片 | 把最终图片变成带字段和图层的模板 | 锁定底图 + 文字层 + 遮盖层 |
| 高频改日期和地点 | Photoshop、Figma 功能强但学习成本高 | 右侧字段面板直接改常用变量 | 字段绑定文字层 |
| 遮住旧文字 | 普通图片编辑器需要手动涂抹 | 提供矩形遮盖层并可吸取底图颜色 | MaskLayer + 取色工具 |
| Prompt 溯源 | Prompt 分散在聊天记录或笔记 | Prompt 跟模板一起保存 | PromptInfo 元数据 |
| 本地备份迁移 | 浏览器工具常只存当前设备 | 可导出模板 JSON，包含底图资产 | JSON + base64 asset |

说明：这里的竞品判断基于常见工作流推断，未做实时竞品功能验证。

### 1.2 三类用户画像
| 角色 | 核心目标 | 对现有工具最大的不满 | 让他们愿意切换的那一个功能 |
|---|---|---|---|
| 课程运营负责人 | 每天或每周复用课程节奏、课程表、学习安排图 | 课程时间、主题、讲师改动频繁，手工改图容易漏 | 课程表字段化，一次改时间、主题、讲师 |
| 活动增长运营 | 复用启动仪式、收益宣传、签到送礼等活动海报 | 同一版式反复改日期、数字、团名和权益信息 | 打开模板后只改活动字段并导出 |
| AI 生图重度用户 | 保存高质量成品图和生成说明 | 成品图、Prompt、改字版本分散 | 图片、Prompt、版本统一归档 |

### 1.3 可行性边界
| 在范围内（及原因） | 明确排除在外（及原因） |
|---|---|
| 桌面浏览器完整编辑，因为画布、模板库、属性面板需要同时可见 | 移动端编辑不做，因为触控拖拽和三栏工作台会显著增加复杂度 |
| 导入 PNG、JPG、WebP 成品图作为底图，因为这是模板复用的起点 | 不调用 AI 模型重新生图，因为本产品解决复用和改字，不解决生成 |
| 文字层、矩形遮盖层、字段绑定，因为它们覆盖最常见的改字需求 | 不做钢笔路径、复杂蒙版、滤镜和图像修复，因为会变成修图软件 |
| 本地保存模板、图片、版本，因为用户不需要账号即可完成闭环 | 不做云同步、团队协作和权限系统，因为会拖慢桌面单用户 MVP |
| PNG、JPG、JSON 备份和批量 ZIP 可以作为产品输出 | 不做模板市场和公开分享页，因为当前目标是自用生产工具 |
| 使用项目根目录常用图片作为第一批模板样本，因为它们代表真实复用场景 | Web 应用不能自动读取用户电脑目录，必须通过用户上传或内置示例资产导入 |

### 1.4 约束分层
| 硬约束 | 推荐默认 | 发挥空间 |
|---|---|---|
| 桌面端优先，不做移动端布局和移动端测试 | 小屏可提示使用桌面端，但不承诺编辑能力 | 后续可单独做只读或轻量导出模式 |
| 不上传用户图片，不要求账号 | 使用 IndexedDB 保存模板和图片资产 | 可增加更明显的备份提醒 |
| 预览和导出必须来自同一份图层数据 | 预览、单张导出、批量导出共用渲染逻辑 | 可增加导出前校验和缩略图预览 |
| P0 只做模板复用，不做重新生图 | Prompt 作为溯源信息保存 | 可增加复制 Prompt、复制负面词 |
| 不把压平旧文字宣称为可直接编辑 | 用遮盖层盖旧字，再添加新文字层 | 可加入 OCR 辅助定位旧字，但不进当前版本 |
| 常用图必须能按用途归类 | 默认归为横版海报、竖版海报、课程表、导师介绍、宣誓证书五类 | 可根据文件名或用户标签自动建议分类 |

### 1.5 当前素材样本观察
根目录当前可复用图片样本约 14 张，另有页面测试截图不应进入模板样本。可复用图片的尺寸和用途呈现出明显模式：

| 样本类型 | 典型尺寸 | 代表内容 | 高频可变字段 |
|---|---:|---|---|
| 横版收益或活动海报 | 1448x1086 | 未来价值操盘团、65% 建仓收益、签到送礼 | 团名、收益数字、日期、权益文案、讲师头像或姓名 |
| 横版课程表 | 1448x1086 | 每日课程节奏、时间段课程安排 | 时间、课程标题、讲师、每日主题、图标状态 |
| 竖版启动仪式海报 | 983x1601、1086x1448 | 启动仪式、活动倒计时、品牌主标题 | 活动标题、日期、团名、地点、底部信息 |
| 导师或人物介绍 | 1054x1492 | 讲师照片、姓名、简介、身份背书 | 姓名、头衔、简介、课程主题 |
| 宣誓词或证书类 | 1086x1448、1448x1086 | 宣誓词、奖章、证书、仪式感页面 | 标题、姓名、宣言文本、日期 |

产品决策：P0 不需要自动把这些根目录图片全部导入系统，但创建模板时应围绕这些样本给出默认分类和默认字段建议。实现层面可以先让用户手动上传图片，再在创建弹窗中选择或自动建议模板类型。

## 第四章：超越竞品的差异化功能

### 第 4.1 节 压平成品图的字段化复用
1. 竞品为何没有这个功能：此处未验证具体竞品现状。基于常见工作流推断，设计软件通常以自由图层编辑为中心，AI 生图工具通常以重新生成图片为中心，两者都不优先服务“成品图只改几个字”的重复生产场景。
2. 本产品如何实现：将底图锁定为不可误删背景，再用遮盖层处理旧文字区域，用文字层和字段表单表达可变内容。
3. 交互流程：
```text
导入成品海报
    |
    v
双击旧文字区域添加遮盖
    |
    v
添加新文字层并绑定字段
    |
    v
保存模板
    |
    v
下次只改字段并导出
```
4. 风险与应对：复杂背景上遮盖层可能突兀；P0 用取色缓解，P2 再考虑 OCR 或 AI 修复。

### 第 4.2 节 本地优先且可迁移的模板资产
1. 竞品为何没有这个功能：不少在线编辑器把资产和账号、云空间绑定，用户迁移依赖平台；本工具的结构选择是单用户本地生产。
2. 本产品如何实现：图片 blob 存 IndexedDB，模板 JSON 导出时内嵌 asset base64，便于在另一台电脑恢复。
3. 交互流程：
```text
用户完成模板
    |
    v
点击导出模板 JSON
    |
    v
拿到内含底图和模板数据的备份
    |
    v
另一浏览器导入
    |
    v
恢复同一模板
```
4. 风险与应对：JSON 文件可能较大；P0 接受，P1 可提供 ZIP 备份或压缩策略。

### 第 4.3 节 Prompt 作为模板记忆而不是生成入口
1. 竞品为何没有这个功能：生图工具侧重生成新结果，设计工具侧重视觉编辑，Prompt 往往不会和最终模板的字段化编辑绑定。
2. 本产品如何实现：Prompt 面板只记录模型、Prompt、负面 Prompt 和风格备注，不触发任何生成请求。
3. 交互流程：
```text
创建模板时填写 Prompt
    |
    v
模板保存 PromptInfo
    |
    v
后续打开模板查看或复制
    |
    v
必要时手动去生图工具复用
```
4. 风险与应对：用户可能误解为可重新生成；界面文案应写明“仅记录，不参与渲染”。

核心设计决策：模板以图层和字段为中心，而不是以页面或文件夹为中心；图片资产和模板元数据分离，便于本地保存和 JSON 迁移；模板类型和字段预设只帮助用户起步，不强制决定最终字段；Prompt 不参与渲染，避免让产品边界滑向生图平台。

## 第六章：技术架构
```text
+----------------------------------------------------------------+
| React UI 层：顶栏、模板库、画布容器、右侧面板、导出弹窗         |
| 职责：展示状态，接收用户操作，提供桌面工作台界面                |
+----------------------------------------------------------------+
| Zustand 状态层：currentTemplate、selectedLayerId、undo、redo    |
| 职责：维护编辑状态，保证字段和图层同步                          |
+----------------------------------------------------------------+
| Fabric 渲染层：屏幕预览、文字层、遮盖层、图片层、导出画布       |
| 职责：用同一份 layers 数据生成预览和输出                        |
+----------------------------------------------------------------+
| Dexie 存储层：templates 表、assets 表                           |
| 职责：本地保存模板和图片 blob，不依赖账号或服务器               |
+----------------------------------------------------------------+
| 文件输出层：PNG、JPG、JSON、ZIP                                 |
| 职责：把本地模板转成用户可下载的文件                            |
+----------------------------------------------------------------+
```

| 库名 | 用途 | 为何优于替代方案 | 大致包体积 |
|---|---|---|---|
| React | 构建桌面工具界面 | 项目已使用，组件拆分清晰 | 未知 |
| Vite | 本地开发和构建 | 项目已使用，启动和构建配置简单 | 未知 |
| Fabric.js | 画布对象、选中、拖拽、导出 | 比原生 Canvas 少写大量图层编辑逻辑 | 未知 |
| Zustand | 当前模板、撤销栈、选中状态 | 状态 API 简洁，适合单页工具 | 未知 |
| Dexie | IndexedDB 封装 | 适合保存 blob 和结构化模板 | 未知 |
| JSZip | 批量导出打包 | 复用成熟 ZIP 生成能力 | 未知 |

最大架构风险：预览、单张导出、批量导出如果分叉实现，会出现用户看到的画布和下载文件不一致。应对方式：所有输出必须调用同一套模板渲染函数，差异只允许在格式和字段覆盖值上出现。

可替换技术原则：推荐沿用现有 React + Vite + Fabric.js + Zustand + Dexie。若未来迁移到 Konva 或 Canvas 2D，可以替换渲染层；不可变的是模板数据模型、字段到文字层的绑定关系、本地备份能力，以及预览和导出同源。

## 第八章：导出与输出系统

### 8.1 支持的输出格式
| 格式 | 使用场景 | 质量选项 | 备注 |
|---|---|---|---|
| PNG | 正式海报、社媒封面、需要清晰文字的图片 | 固定无损 | 当前必须稳定 |
| JPG | 文件更小的传播图 | 80%、90%、100% | 当前已支持 |
| JSON | 模板备份和跨设备迁移 | 完整模板和图片资产 | 当前必须稳定 |
| ZIP | 批量导出多期图片 | 图片 + 可选 JSON | 当前可作为增强能力 |

### 8.3 批量处理流程
```text
用户打开批量导出
    |
    v
系统以当前字段生成第一行变量
    |
    +-- 用户添加多期内容
            |
            v
       校验每行必填字段
            |
            +-- 有空值 --> 阻止导出并提示第几期哪个字段
            |
            +-- 全部通过 --> 串行渲染每张图片
                                |
                                v
                           加入 ZIP
                                |
                                v
                           下载 ZIP
```

批量渲染不并行执行，避免多张大图同时占用内存。

## 第九章：开发优先级
- P0 - 没有这个，产品根本无法使用。交付标准：桌面端打开应用，从常用素材导入一张图，选择或填写模板类型，创建模板，编辑字段和图层，保存，重新打开，单张导出，导出模板备份。
- P1 - 没有这个，用户第一次体验后不会回来。交付标准：首屏空态清楚，右侧无模板状态清楚，模板类型筛选可用，默认字段预设可用，撤销重做稳定，批量导出可用，版本恢复可用。
- P2 - 有了这个，用户会把产品推荐给别人。交付标准：适应画布缩放，缩略图模板库，字段类型，导出前预览，OCR 辅助定位文字区域。
- P3 - 有了这个，一部分用户会付费或强烈倡导。交付标准：云同步，团队共享，模板权限，AI 擦除旧文字，插件化素材层。

## 第十一章：开发者交接说明

### a) 实现顺序建议
你先保护桌面端三栏工作台的主流程，再做视觉空态。优先顺序是：模板创建和保存、字段和图层联动、导出同源渲染、首次空态、右侧无模板说明、备份风险提示。原因是这个产品的价值不在页面数量，而在同一份模板数据能被创建、恢复、编辑和导出。

### b) 最可能导致返工的三个决策
| 决策是什么 | 安全的默认选择是什么 | 什么信号提示你需要改变方向 |
|---|---|---|
| 是否做移动端 | 不做移动端编辑，只做桌面端 | 用户明确要求手机创建和导出，且愿意接受单独设计 |
| 是否接入 AI 重新生图 | 不接入，只保存 Prompt | 用户把“重新生成同款图”列为核心付费理由 |
| 是否做云同步账号 | 不做账号，本地保存加 JSON 备份 | 用户需要多设备长期同步或团队共享 |

### c) 哪里要严格，哪里可以灵活
| 章节 | 约束或建议 | 原因 |
|---|---|---|
| 第一章产品边界 | 约束 | 防止偏成移动端、营销页或生图平台 |
| 第二章布局 | 约束 | 三栏桌面工作台是当前核心效率来源 |
| 第三章模块 | 约束 | 每个模块都服务创建、编辑、保存、导出闭环 |
| 第四章差异化 | 建议 | 可以换表达，但不能丢掉字段化复用 |
| 第五章数据模型 | 约束 | 数据结构决定能否长期恢复模板 |
| 第六章技术架构 | 建议 | 可沿用现有栈，但预览和导出必须同源 |
| 第七章交互细节 | 建议 | 可以优化文案和状态，但不要扩大功能面 |
| 第八章输出 | 约束 | PNG 和 JSON 备份必须稳定 |

### d) 已知的未知项
- 此处已观察：当前常用图片包含 1448x1086 横版图、1086x1448 和 1024x1536 竖版图、983x1601 长图；后续仍需确认这些是否代表长期素材分布。
- 此处未解决：复杂背景上遮盖旧文字的视觉质量是否足够，默认先用取色和手动调整解决。
- 此处未解决：批量导出的真实上限未知，默认用串行渲染降低内存风险。
- 此处未验证：具体竞品是否已经支持类似字段化复用，当前差异化基于常见工作流推断。
